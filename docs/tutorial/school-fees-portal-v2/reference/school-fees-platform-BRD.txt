# School Fees & Payments Platform  
**Business Requirements Document (BRD)**

## 1. Purpose & Business Objective

**Purpose**  
To provide a centralized, secure, and configurable platform for institutes to manage fee collection, payments through dedicated admin and user-facing portals.

**Business Objectives**
- Digitize institute fee collection workflows
- Reduce manual errors in payment tracking and reconciliation
- Enable self-service online payments for students
- Provide auditability, security, and role-based access
- Allow institutes to configure and brand their own user frontend

## 2. Stakeholders & Users

**Primary Stakeholders**
- Institute Owners / Management
- Institute Finance & Administration Teams
- Students / Parents or Guardians

**User Roles**
- **Platform Admin** – Manages institutes, institute users and global configuration
- **Institute Admin** – Manages institute data, institute users, institute fee types, and collections and payment gateway settings for the institute.
- **Student (Public User)** – Views dues and makes payments. Can also view the payment history and cancelled payments

## 3. In-Scope Features

### Admin Platform
- Institute onboarding
- Fee type configuration and maintenance
- Institute admin user invitation and management
- Payment collection initiation (bulk via Excel upload)
- Student record creation and management
- Payment monitoring, reconciliation
- Cancellation handling
- (Currently no refunds - future scope)


### User Platform
- Student authentication via email otp
- View pending dues and payment history and cancelled payments
- Online payment initiation and confirmation

## 4. Key Business Use Cases

| Use Case | Role | Outcome |
|--------|------|--------|
| Institute Onboarding | Platform Admin | Institute created and activated |
| Fee Type Setup | Institute Admin | Fee structures configured |
| Admin User Invitation | Institute Admin | Controlled access to institute |
| Initiate Payment Collection | Institute Admin | Bulk payment demands created |
| Student Login | Student | Secure access to dues |
| Make Payment | Student | Payment completed and recorded |
| Cancel Payment | Institute Admin | Revoke initiated payment |
| Configure User Frontend | Institute Admin | Branded student experience |

## 5. Business Rules & Constraints
- Only Platform and Institute admins have access to the admin panel
- Institute admins cannot create new institutes
- Institute admins can only access their own institute data
- Students can only view and pay their own dues by logging into the student portal
- All financial and admin actions must be auditable
- Role-based access governs both data access and UI visibility

## 6. Data Model Overview

### Domain Entities
- Institute
    Contains details about the educational institute such as name, address, contact info, payment gateway settings, support personnel info, branding details, etc.
- Institute User
    Represents users associated with an institute. Currently only institute admins are supported.
- Fee Type
    Represents different fee structures defined by the institute (e.g., tuition, library fee, lab fee).
    Fee types can support full or partial payments.
    Late fee penalty configuration.
- Student
    Represents students enrolled in the institute. Contains student name, login ID and parent/guardian contact information.
- Payment Collection
    Represents a bulk collection event initiated by the institute admin. Contains details about the collection such as Student name, Student contact info, fees due, fees due date for a one or multiple fee types and the payment mode.
- Payment Collection Item
    Represents the payment collection item per student per fee type as part of a payment collection. If a payment collection includes multiple fee types for a student, there will be multiple payment collection items created for that student.
- Payment Collection Item Detail
    Represents the payment details for a payment collection item. This includes the amount paid, payment date, payment status. In case of partial payments, multiple payment collection item details will be created for a payment collection item. 
- Payment
    Represents the actual payment made by the student. Contains payment gateway transaction details, payment status, amount paid, payment date, pg order id, pg transaction id, etc. A single payment can cover multiple payment collection item details.
### Relationships
- One Institue -> Many Institute Users
- One Institute -> Many Fee Types
- One Institute -> Many Students
- One Payment Collection -> Many Payment Collection Items
- One Payment Collection Item -> Many Payment Collection Item Details
- One Payment -> Many Payment Collection Item Details

## 7. Non-Functional Requirements

**Security**
- Role-based access control (RBAC)
- Data isolation per institute
- Audit trail for critical operations

**Performance**
- Background processing for bulk uploads
- Asynchronous email and notification delivery

**Compliance**
- Secure payment processing
- Transaction traceability and reporting

## 8. Assumptions & Dependencies

**Assumptions**
- Institutes maintain accurate student records
- Students / Parents or Guardians have access to email
- Online payment gateways are available and supported

**Dependencies**
- Email (SMTP) service
- Payment gateway integration
