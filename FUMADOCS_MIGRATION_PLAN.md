# Docusaurus to Fumadocs Migration Plan

## Goal

Migrate the SolidX documentation site from Docusaurus to Fumadocs while preserving the existing public documentation URLs, content hierarchy, and image/download assets. Do **not** preserve the current Docusaurus styling or custom SolidX documentation theme; use Fumadocs' default styling and components as the visual baseline.

Target state:

- Next.js App Router site using Fumadocs UI and Fumadocs MDX.
- Documentation available under the current `/docs/...` URL space.
- Existing Markdown/MDX content migrated into Fumadocs' `content/docs` convention.
- Existing sidebar order reproduced through Fumadocs `meta.json` files.
- Existing static asset URLs such as `/img/...` and `/downloads/...` continue to work.
- Default Fumadocs UI styling used for docs layout, typography, sidebar, callouts, tabs, search, dark/light mode, and code blocks.
- Docusaurus-specific packages, config, aliases, and theme components removed after parity is verified.

## Current Site Inventory

### Framework and build

Current stack is Docusaurus 3.7.0 with the classic preset.

Important files:

- `package.json` — Docusaurus scripts and dependencies.
- `docusaurus.config.ts` — site metadata, navbar/footer, Prism, sitemap, custom CSS, commented Typesense search config.
- `sidebars.ts` — single autogenerated sidebar from `docs/`.
- `tsconfig.json` — extends `@docusaurus/tsconfig` and only defines `baseUrl`.
- `src/pages/index.tsx` — Docusaurus homepage.
- `src/theme/MDXComponents.jsx` — overrides `img` to use `react-medium-image-zoom`; do not carry this forward unless image zoom is explicitly required later.
- `src/css/custom.css` — large custom SolidX theme and documentation styles; do not port this into the Fumadocs site.

### Content

- `docs/` contains 145 Markdown/MDX documentation files.
- Top-level doc sections:
  - `tutorial/`
  - `admin-docs/`
  - `developer-docs/`
  - `recipes/`
- `blog/` contains 4 Docusaurus starter/sample blog files. The blog navbar entry is currently commented out, so decide whether to migrate or delete/ignore it separately from the docs migration.
- `static/` contains 179 files, including:
  - 163 PNG images.
  - 9 OTF fonts.
  - 4 SVG files.
  - 1 JSON download.
  - 1 JPG.
  - 1 ICO.

### Frontmatter

Observed docs frontmatter fields:

- `sidebar_position` in 138 files.
- `title` in 119 files.
- `description` in 81 files.
- `summary` in 74 files.
- `keywords` in 24 files.
- Metadata-schema-specific fields such as `json_pointer`, `jsonpath`, `parent_component`, `items_type`, and `items_attributes_doc`.
- 26 docs are missing `title` frontmatter and should be normalized during migration.

Fumadocs uses `title`, `description`, and optional `icon` directly for page tree generation. Other fields can remain in frontmatter if the Fumadocs MDX schema allows them.

### Docusaurus-specific dependencies in content

Known migration hotspots:

- Docusaurus aliases/imports:
  - `@site/src/common/*`
  - `@site/src/components/*`
  - `@theme/Tabs`
  - `@theme/TabItem`
  - `@theme/Heading`
  - `@docusaurus/*`
- Docusaurus admonition syntax:
  - `:::tip`
  - `:::note`
  - `:::info`
  - `:::warning`
  - `:::danger SECURITY WARNING`
- Custom Docusaurus MDX image zoom override in `src/theme/MDXComponents.jsx`.
- Absolute static asset links, e.g. `/img/...` and `/downloads/...`.
- Custom JSX/classes in docs such as `card-headear-wrapper`, `tips-box`, `white-color`, `rotatable`, and Docusaurus grid classes (`container`, `row`, `col`). These should be removed or converted to default Fumadocs/semantic Markdown patterns rather than reimplemented.

### Styling decision

The migration should intentionally discard the current Docusaurus visual layer:

- Do not port `src/css/custom.css` into `app/globals.css`.
- Do not recreate Docusaurus navbar/sidebar/footer styling.
- Do not preserve custom docs classes such as `tips-box`, `card-headear-wrapper`, `white-color`, `rotatable`, `container`, `row`, or `col`.
- Do not port custom callout box components for visual parity.
- Do not preserve decorative `react-icons` usage in MDX headings/cards just to match the old look; convert those sections to semantic Markdown or default Fumadocs components.
- Prefer Fumadocs' default MDX components, default `Callout`, default tabs, default typography, default sidebar, default search dialog, and default code block styling.
- Keep only minimal global CSS required by Fumadocs itself.

## Fumadocs Conventions to Adopt

Use Fumadocs framework mode with Fumadocs MDX:

- Content source: `content/docs`.
- Page routing handled by Next.js route `app/docs/[[...slug]]/page.tsx`.
- Fumadocs source loader in `lib/source.ts` with `baseUrl: '/docs'`.
- Sidebars/page tree generated from content plus `meta.json` files.
- `meta.json` `pages` arrays control folder ordering. If `pages` is present, unlisted pages are excluded, so generated files should include `...` only when intentionally allowing remaining alphabetic pages.
- Folder `index.md`/`index.mdx` is the clickable index page by default.
- Root folders can use `root: true` to become Fumadocs UI layout tabs.

Recommended Fumadocs shape:

```text
app/
  layout.tsx
  page.tsx
  docs/
    layout.tsx
    [[...slug]]/
      page.tsx
  api/
    search/
      route.ts
components/
  mdx.tsx
content/
  docs/
    meta.json
    tutorial/
      index.md
      meta.json
    admin-docs/
      index.md
      meta.json
    developer-docs/
      index.md
      meta.json
    recipes/
      index.md
      meta.json
lib/
  source.ts
  layout.shared.tsx
public/
  img/
  downloads/
  fonts/
source.config.ts
next.config.mjs
```

## Migration Strategy

Use a staged migration instead of editing all content by hand. The docs set is large enough that automated copying, metadata generation, and targeted content transforms are safer.

### Phase 1 — Bootstrap Next.js + Fumadocs

1. Replace Docusaurus runtime with Next.js + Fumadocs dependencies.

   Install/add:

   ```bash
   npm install next fumadocs-ui fumadocs-core fumadocs-mdx @types/mdx
   ```

   Keep existing dependencies that are still used by content/components:

   - `react`
   - `react-dom`
   - `prismjs`/`prism-react-renderer` only if still required after Fumadocs code highlighting is configured

   Remove `react-icons` and `react-medium-image-zoom` unless either is later required for a non-styling feature.

2. Update `package.json` scripts:

   ```json
   {
     "scripts": {
       "dev": "next dev -p 3003",
       "start": "next start -p 3003",
       "build": "next build",
       "typecheck": "tsc --noEmit"
     }
   }
   ```

   Optionally keep `docusaurus:*` scripts temporarily during migration, but remove them before final cutover.

3. Add `next.config.mjs`:

   ```js
   import { createMDX } from 'fumadocs-mdx/next';

   /** @type {import('next').NextConfig} */
   const config = {
     reactStrictMode: true,
   };

   const withMDX = createMDX();

   export default withMDX(config);
   ```

4. Add `source.config.ts`:

   ```ts
   import { defineConfig, defineDocs } from 'fumadocs-mdx/config';

   export const docs = defineDocs({
     dir: 'content/docs',
   });

   export default defineConfig();
   ```

5. Replace `tsconfig.json` inheritance from Docusaurus with a Next-compatible config and add Fumadocs generated source alias:

   ```json
   {
     "compilerOptions": {
       "target": "ES2017",
       "lib": ["dom", "dom.iterable", "esnext"],
       "allowJs": true,
       "skipLibCheck": true,
       "strict": true,
       "noEmit": true,
       "esModuleInterop": true,
       "module": "esnext",
       "moduleResolution": "bundler",
       "resolveJsonModule": true,
       "isolatedModules": true,
       "jsx": "preserve",
       "incremental": true,
       "baseUrl": ".",
       "paths": {
         "@/*": ["./*"],
         "collections/*": ["./.source/*"]
       }
     },
     "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx", ".next/types/**/*.ts", ".source/**/*.ts"],
     "exclude": ["node_modules"]
   }
   ```

### Phase 2 — Add Fumadocs app shell

1. Add `app/layout.tsx`:

   ```tsx
   import './globals.css';
   import { RootProvider } from 'fumadocs-ui/provider/next';
   import type { ReactNode } from 'react';

   export default function Layout({ children }: { children: ReactNode }) {
     return (
       <html lang="en" suppressHydrationWarning>
         <body className="flex min-h-screen flex-col">
           <RootProvider>{children}</RootProvider>
         </body>
       </html>
     );
   }
   ```

2. Add `app/globals.css`:

   ```css
   @import 'tailwindcss';
   @import 'fumadocs-ui/css/neutral.css';
   @import 'fumadocs-ui/css/preset.css';
   ```

   Do not import or copy `src/css/custom.css`. Add only genuinely required global resets or font declarations if Fumadocs requires them later.

3. Add `lib/source.ts`:

   ```ts
   import { docs } from 'collections/server';
   import { loader } from 'fumadocs-core/source';

   export const source = loader({
     baseUrl: '/docs',
     source: docs.toFumadocsSource(),
   });
   ```

4. Add `lib/layout.shared.tsx` for shared navigation options:

   ```tsx
   import type { BaseLayoutProps } from 'fumadocs-ui/layouts/shared';

   export function baseOptions(): BaseLayoutProps {
     return {
       nav: {
         title: 'SolidX Docs',
       },
       links: [
         { text: 'Tutorial', url: '/docs/tutorial' },
         { text: 'Admin Docs', url: '/docs/admin-docs' },
         { text: 'Developer Docs', url: '/docs/developer-docs' },
         { text: 'Recipes', url: '/docs/recipes' },
       ],
     };
   }
   ```

5. Add `app/docs/layout.tsx`:

   ```tsx
   import { source } from '@/lib/source';
   import { DocsLayout } from 'fumadocs-ui/layouts/docs';
   import { baseOptions } from '@/lib/layout.shared';

   export default function Layout({ children }: LayoutProps<'/docs'>) {
     return (
       <DocsLayout tree={source.getPageTree()} {...baseOptions()}>
         {children}
       </DocsLayout>
     );
   }
   ```

6. Add `app/docs/[[...slug]]/page.tsx`:

   ```tsx
   import { source } from '@/lib/source';
   import { getMDXComponents } from '@/components/mdx';
   import { createRelativeLink } from 'fumadocs-ui/mdx';
   import {
     DocsBody,
     DocsDescription,
     DocsPage,
     DocsTitle,
   } from 'fumadocs-ui/layouts/docs/page';
   import type { Metadata } from 'next';
   import { notFound } from 'next/navigation';

   export default async function Page(props: PageProps<'/docs/[[...slug]]'>) {
     const params = await props.params;
     const page = source.getPage(params.slug);
     if (!page) notFound();

     const MDX = page.data.body;

     return (
       <DocsPage toc={page.data.toc} full={page.data.full}>
         <DocsTitle>{page.data.title}</DocsTitle>
         <DocsDescription>{page.data.description}</DocsDescription>
         <DocsBody>
           <MDX components={getMDXComponents({ a: createRelativeLink(source, page) })} />
         </DocsBody>
       </DocsPage>
     );
   }

   export function generateStaticParams() {
     return source.generateParams();
   }

   export async function generateMetadata(
     props: PageProps<'/docs/[[...slug]]'>,
   ): Promise<Metadata> {
     const params = await props.params;
     const page = source.getPage(params.slug);
     if (!page) notFound();

     return {
       title: page.data.title,
       description: page.data.description,
     };
   }
   ```

7. Add `app/api/search/route.ts` for built-in Orama search:

   ```ts
   import { source } from '@/lib/source';
   import { createFromSource } from 'fumadocs-core/search/server';

   export const { GET } = createFromSource(source, {
     language: 'english',
   });
   ```

### Phase 3 — Move assets and preserve URLs

1. Move Docusaurus `static/*` to Next.js `public/*`:

   ```text
   static/img/*       -> public/img/*
   static/downloads/* -> public/downloads/*
   static/fonts/*     -> public/fonts/*
   ```

2. Keep paths unchanged in Markdown where possible. Existing links like:

   ```md
   ![Global Search](/img/admin-docs/modules/list-view-global-search.png)
   ```

   should continue to work after `static/img` becomes `public/img`.

3. Verify referenced assets exist. Add a script/check that extracts `/img/...` and `/downloads/...` references and checks them against `public/`.

4. Update favicon/social images:

   - Current config references `img/tab-logo.png`, but inventory shows `static/img/favicon.ico`; verify whether `tab-logo.png` exists.
   - Add Next.js metadata in `app/layout.tsx` for title, description, icons, and OpenGraph images.

### Phase 4 — Copy and normalize docs content

1. Copy `docs/` to `content/docs/` preserving relative paths.

2. Keep `index.md` files in place. Fumadocs will use them as folder index pages.

3. Preserve the `/docs/...` URL structure. With `baseUrl: '/docs'` and `content/docs/admin-docs/index.md`, the path remains `/docs/admin-docs`.

4. Normalize frontmatter:

   - Keep `title` and `description`.
   - Keep `summary`, `keywords`, and custom metadata fields only if the Fumadocs MDX schema is configured to allow them.
   - Remove or ignore `sidebar_position` after generating `meta.json` files.
   - Add missing `title` values to the 26 files that do not currently define one. Derive titles from the first `# Heading`, then from filename if no heading exists.

5. Consider excluding scratch/planning content from published docs unless intentionally public:

   - `docs/tutorial/school-fees-portal-v2/tmp/*`

   If retained, give the folder a clear title and add it deliberately to its folder `meta.json`.

### Phase 5 — Generate Fumadocs `meta.json` files

Because Docusaurus currently autogenerates the sidebar and uses `sidebar_position`, generate one `meta.json` per folder to make Fumadocs ordering deterministic.

Generation rules:

1. For each folder under `content/docs`, list direct page/folder children.
2. Use each child's `sidebar_position` from its `index.md` or file frontmatter when present.
3. Sort by numeric `sidebar_position`; support decimals such as `3.5`.
4. For ties or missing positions, sort alphabetically by title/slug.
5. Write only slugs relative to the current folder, without extension:

   ```json
   {
     "title": "Developer Docs",
     "pages": [
       "index",
       "prerequisites",
       "installation",
       "project-structure",
       "database-seeding",
       "extending",
       "rest-apis",
       "metadata_schema",
       "going-live"
     ]
   }
   ```

6. Root `content/docs/meta.json` should make the main areas explicit. Recommended first pass:

   ```json
   {
     "title": "SolidX Docs",
     "pages": [
       "tutorial",
       "admin-docs",
       "developer-docs",
       "recipes"
     ]
   }
   ```

7. Decide whether to mark top-level folders as Fumadocs roots:

   - Option A, recommended for parity with one global Docusaurus sidebar: do **not** set `root: true`; show one combined sidebar.
   - Option B, recommended if you want Fumadocs layout tabs: set `root: true` on `tutorial`, `admin-docs`, `developer-docs`, and `recipes`. This changes navigation behavior by showing only the selected root folder's items.

### Phase 6 — Replace Docusaurus MDX behavior

1. Add `components/mdx.tsx` and keep it close to the Fumadocs default:

   ```tsx
   import defaultMdxComponents from 'fumadocs-ui/mdx';
   import type { MDXComponents } from 'mdx/types';

   export function getMDXComponents(components?: MDXComponents) {
     return {
       ...defaultMdxComponents,
       ...components,
     } satisfies MDXComponents;
   }

   export const useMDXComponents = getMDXComponents;

   declare global {
     type MDXProvidedComponents = ReturnType<typeof getMDXComponents>;
   }
   ```

2. Do not port custom callout components from `src/common/*`. Replace their usage with Fumadocs' default `Callout` component or plain Markdown.

3. Replace Docusaurus alias imports in docs:

   ```diff
   - import { NoteBoxs } from '@site/src/common/NoteBoxs';
   + import { Callout } from 'fumadocs-ui/components/callout';
   ```

   Prefer registering `Callout` globally in `components/mdx.tsx` if repeated imports become noisy. Do not recreate `NoteBoxs`, `InfoBox`, `WarningBox`, `TipBox`, `DangerBox`, or `SuccessBox` for visual parity.

4. Convert Docusaurus admonitions to Fumadocs-compatible JSX. Example:

   ```diff
   - :::tip For the Fast Track: Model Metadata
   - Content
   - :::
   + <Callout type="info" title="For the Fast Track: Model Metadata">
   + Content
   + </Callout>
   ```

   Confirm exact `Callout` props against the installed Fumadocs UI version. If the default component API differs, adjust to the Fumadocs-supported API rather than adding custom wrappers styled like the old site.

5. Replace `@theme/Tabs` and `@theme/TabItem` with Fumadocs tabs or plain headings. There are only a few Docusaurus tab imports, so manual conversion is acceptable.

6. Replace Docusaurus `Heading`, `Link`, `Layout`, and `useDocusaurusContext` usages from homepage/components with Next.js/Fumadocs equivalents.

7. Do not preserve the Docusaurus image zoom override by default. Use Fumadocs' default image rendering unless image zoom is later requested as a separate feature.

### Phase 7 — Rebuild homepage and navigation with default Fumadocs styling

1. Convert `src/pages/index.tsx` to `app/page.tsx`.

2. Replace Docusaurus components:

   - `@docusaurus/Link` -> `next/link`.
   - `@theme/Layout` -> Next.js root layout and page metadata.
   - `@theme/Heading` -> semantic `h1`, `h2`, etc.
   - `useDocusaurusContext()` -> constants in a local config file or Next metadata.

3. Convert `src/components/HomepageFeatures` and `src/components/Tutorials`:

   - Replace `require('@site/static/img/...').default` with either `/img/...` strings or Next `Image` imports.
   - Do not preserve Docusaurus grid classes or existing card styling. Rebuild homepage sections using simple semantic markup and Fumadocs/default Tailwind utility patterns only where necessary.

4. Do not port `src/css/custom.css`.

   - Delete or ignore old design tokens such as `--solid-primary`, `--solid-bg-*`, and `--solid-text-*` unless a minimal brand color is explicitly requested later.
   - Remove custom content classes from docs while converting content.
   - Let Fumadocs own typography, sidebar, callouts, search, code blocks, and dark/light mode.
   - Add only the minimal CSS needed for functional layout gaps that Fumadocs does not cover.

5. Configure navbar links through `baseOptions()` using Fumadocs' default layout. Avoid custom header/footer components unless a required navigation item cannot be represented by Fumadocs defaults.

### Phase 8 — Search migration

Current Docusaurus Typesense config is commented out, so the lowest-risk migration is Fumadocs' built-in Orama search first.

1. Enable the `/api/search` route from Phase 2.
2. Confirm search opens with `Cmd/Ctrl+K` through `RootProvider` default behavior.
3. If Typesense must remain, implement a custom `SearchDialog` and pass it to `RootProvider`:

   ```tsx
   <RootProvider
     search={{
       SearchDialog: SolidXTypesenseSearchDialog,
     }}
   >
     {children}
   </RootProvider>
   ```

4. Only port `docusaurus-theme-search-typesense`, `typesense`, and `typesense-instantsearch-adapter` concepts after built-in search parity is verified. Do not carry Docusaurus search packages into the final build.

### Phase 9 — Blog decision

The current blog appears to be Docusaurus starter/sample content and is not linked in the navbar.

Choose one:

- **Recommended:** Do not migrate blog in the first docs migration. Remove sample blog content after stakeholder confirmation.
- If blog is required, create a separate Fumadocs MDX collection in `source.config.ts` with `dir: 'content/blog'`, build `app/blog` routes, and map existing blog frontmatter (`slug`, `title`, `authors`, `tags`) into the new blog schema.

### Phase 10 — Remove Docusaurus

After Fumadocs builds and parity checks pass:

1. Delete Docusaurus-only files:
   - `docusaurus.config.ts`
   - `sidebars.ts`
   - `.docusaurus/` if present
   - `src/theme/MDXComponents.jsx`
2. Remove Docusaurus dependencies:
   - `@docusaurus/core`
   - `@docusaurus/preset-classic`
   - `@docusaurus/plugin-sitemap`
   - `@docusaurus/module-type-aliases`
   - `@docusaurus/tsconfig`
   - `@docusaurus/types`
   - `docusaurus-theme-search-typesense` if not replaced by a custom Fumadocs search dialog
3. Remove Docusaurus scripts from `package.json`.
4. Update `README.md` with Next.js/Fumadocs commands and deployment instructions.
5. Update deployment config if it assumes Docusaurus `build/`; Next.js output differs unless configured for static export.

## Automation Needed

Create temporary migration scripts and remove them after successful migration, or keep them under `scripts/` only if they remain useful for future content audits.

Recommended scripts/checks:

1. `copy-docs-to-content`:
   - Copies `docs/` to `content/docs/`.
   - Does not overwrite manual edits without confirmation.

2. `generate-fumadocs-meta`:
   - Reads frontmatter from each folder's direct children.
   - Generates `meta.json` files with `pages` ordered by `sidebar_position`.
   - Derives folder titles from folder `index.md` title or folder name.

3. `normalize-frontmatter`:
   - Adds missing `title` values.
   - Optionally removes `sidebar_position` after meta files are generated.
   - Ensures custom fields are valid according to the Fumadocs MDX schema.

4. `convert-admonitions`:
   - Converts Docusaurus `:::` blocks to chosen Fumadocs/local callout syntax.
   - Leaves complex titled admonitions for manual review if parsing is ambiguous.

5. `audit-mdx-imports`:
   - Finds live MDX import lines outside fenced code blocks.
   - Flags imports from `@site`, `@theme`, and `@docusaurus`.
   - Distinguishes content code examples from actual MDX imports.

6. `check-public-assets`:
   - Extracts `/img/...`, `/downloads/...`, and `/fonts/...` references.
   - Verifies each exists under `public/`.

   - Sidebar order matches the old Docusaurus sidebar.
   - Page titles/descriptions render correctly.
   - Table of contents renders on long pages.
   - Code blocks render with expected languages (`typescript`, `tsx`, `bash`, `json`, `http`, `diff`).

   - Tabs render correctly where used.

## Risks and Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Fumadocs excludes pages not listed in `meta.json.pages` | Missing docs in sidebar/tree | Generate comprehensive `pages` arrays or include `...` intentionally. |
| Docusaurus MDX aliases fail | Build errors | Register global MDX components and remove/replace Docusaurus imports. |
| Old content depends on custom CSS classes | Unstyled or awkward content blocks | Convert those blocks to semantic Markdown or default Fumadocs components; do not port the old CSS. |
| Static assets move from `static` to `public` | Broken images/downloads | Preserve URL paths and run an asset-reference checker. |
| Existing docs without `title` | Fumadocs page tree has poor labels/build warnings | Derive titles from first H1 or filename. |
| Scratch `tmp` docs become public | Unwanted content exposure | Exclude or explicitly label the folder before migration. |
| Search behavior changes from Typesense to Orama | Different search ranking/features | Start with Orama; add custom Typesense dialog only if required. |
| Deployment assumes static Docusaurus `build/` | Broken deploy | Decide whether to use Next server output or `output: 'export'`, then update `deploy.sh`, Docker, and hosting config. |

## Suggested Implementation Order

1. Bootstrap Next.js/Fumadocs app with a single sample page.
2. Move static assets to `public/` and verify direct asset URLs.
3. Copy docs into `content/docs/`.
4. Generate and inspect `meta.json` files.
5. Normalize frontmatter and missing titles.
6. Convert Docusaurus MDX imports/admonitions/tabs.
7. Configure default Fumadocs MDX components and remove custom docs styling dependencies.
8. Rebuild homepage/navigation with default Fumadocs styling.
9. Enable search.
10. Run full build and route parity checks.
11. Remove Docusaurus files/dependencies and update README/deployment.

## Definition of Done

- `npm run build` succeeds on the Fumadocs/Next.js app.
- All intended docs pages are available under `/docs/...` with no duplicate URLs.
- Sidebar order is intentionally defined by Fumadocs `meta.json` files.
- No live docs content imports Docusaurus aliases/packages.
- Images and downloads load from `public/` without URL changes.
- Homepage, docs layout, callouts, code blocks, TOC, dark/light theme, and search use Fumadocs defaults and are functional.
- The old `src/css/custom.css`, custom Docusaurus theme overrides, and custom docs visual classes are not carried into the new site.
- Docusaurus packages/config/scripts are removed from the final project.
- README and deployment scripts describe the new Fumadocs workflow.
